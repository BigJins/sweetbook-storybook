## Problem Statement

스위트북 바이브코딩 풀스택 개발자 채용 과제 (마감 4월 29일 23:59).

요구 구조: "콘텐츠 서비스가 본체, 책 만들기는 부가 기능". `api.sweetbook.com`은 호출하지 않고 외부 의존 없이 독립적으로 동작. Docker로 한 방에 실행. Public GitHub. 서술형 4문항 + GitHub URL 구글폼 제출.

선택한 콘텐츠 도메인: **AI 동화 생성** — 아이의 그림과 상상(텍스트 입력)으로 만들어지는 개인화된 동화책 서비스. PDF 예시표의 "AI 콘텐츠 — AI 동화 생성" 카테고리 직격.

## What Makes This Cool

**"내 아이의 그림이 동화책 일러스트가 됐다"** — 입력값(아이의 그림)이 결과물(완성된 책)에 시각적으로 살아있는 게 핵심 wow.

- 아이가 크레용/색연필로 그린 그림을 업로드 → AI가 그 그림의 색감·스타일·터치를 분석
- 아이가 말로 한 상상(예: "곰돌이가 우주에 가서 별을 따왔어")을 텍스트로 입력
- AI가 그 상상을 다섯 페이지짜리 동화로 쓰고, 페이지마다 아이 그림 스타일이 묻어나는 일러스트 생성
- 완성된 동화는 책 미리보기 형태로 열람·편집 가능
- "이 동화를 종이책으로 만들기" 버튼 → 주문 흐름 진입 (Lv2)
- 주문 1건당 콘텐츠 + 메타데이터 ZIP 익스포트 (Lv3)

면접 5분 데모 시나리오: "아이 그림 1장 업로드 + 상상 한 줄 입력 → 30초 안에 동화책 미리보기. before/after를 한 화면에서 보여줌."

## Constraints

- **시간**: 30시간. 마감 직전 6-8시간은 디버깅·Docker 통합·README·시연 영상에 비워둠 → 실제 코딩 가용 시간 약 22-24시간.
- **외부 의존**: OpenAI API 키 1개에 의존. 심사자 환경에서 키 없이도 즉시 데모 가능하도록 더미 데이터 시드 + mock fallback 필요.
- **실행 요건**: `docker-compose up` 한 번에 실행, 포트 환경변수로 변경 가능, README 보고 실행, Public 저장소.
- **기능 미구현**: 결제·배송·회원 인증은 구현 대상 아님. 실제 종이책 인쇄도 아님.
- **PDF 명시 우선순위**: "Lv1 완성도 > Lv3 어설픈 시도".

## Premises (User-Confirmed)

1. AI 동화 카테고리는 PDF 예시표에 명시되어 있어 평가자 친숙성이 높다 → 무난한 선택. ✓
2. "그림(스타일) + 상상(스토리)" 하이브리드가 입력값 두 개를 모두 의미 있게 살리는 동시에 얼굴 보존 같은 고난이도 이미지 생성을 회피한다 → 마감 안전성 + 데모 임팩트의 스윗스팟. ✓
3. Lv1+Lv2+가벼운 Lv3가 평가 3축(서비스 플로우 / 비즈니스 로직 / 데이터 모델링)을 모두 커버하면서 마감 가능. 데이터 모델을 처음부터 잘 잡아두면 Lv3 익스포트 코드는 1-2시간이면 끝남. ✓
4. 사용자 익숙한 스택(Spring Boot 3 + Java 21 + JPA + MySQL + Vite + Vue 3)이 면접 "왜 이 스택?" 답변에 자연스럽고 마감 리스크도 낮다. ✓

## Approaches Considered

### Approach A: 사진 → 신데렐라 주인공 (얼굴 보존)
- Summary: 아이 사진 업로드 → 얼굴을 신데렐라/백설공주 같은 기존 동화 일러스트에 합성.
- Effort: XL / Risk: High
- Pros: 데모 임팩트 최강. "내 아이가 진짜 신데렐라"는 한 줄 설명으로 끝.
- Cons: 얼굴 보존 이미지 생성은 ControlNet/IP-Adapter 또는 Replicate 같은 유료 API 필수. Docker 안에서 안정 동작시키는 데만 8-12h. 30h 마감에 미달 위험.
- Reject 사유: 마감 리스크.

### Approach B: 그림(스타일) + 상상(스토리) 하이브리드 ★ Recommended
- Summary: 아이 그림 = 일러스트 스타일 레퍼런스, 아이 상상 = 스토리 프롬프트.
- Effort: M / Risk: Low-Medium
- Pros: 두 입력 모두 의미 있게 살림. OpenAI 한 키로 LLM(스토리)+이미지(gpt-image-1) 둘 다 처리. Docker 단순.
- Cons: 얼굴 보존만큼의 시각 임팩트는 없음 → before/after 내러티브로 보완.
- Reuses: OpenAI Java SDK 또는 Spring AI starter, Vue 표준 컴포넌트.

### Approach C: 텍스트만 → AI 동화 생성
- Summary: 아이가 텍스트로 상상만 입력, 그림 입력 없음.
- Effort: S / Risk: Very Low
- Pros: 가장 안전. Lv3까지 무조건 끝남.
- Cons: AI 동화 서비스 중에 흔한 형태. "아이 그림"이라는 차별점 못 살림. 5분 데모에 wow 없음.

## Recommended Approach: B (하이브리드)

**핵심 설계**

- **그림 분석 → 스타일 키워드 추출** (간접 방식): GPT-4o Vision이 그림을 보고 "수채화풍, 따뜻한 파스텔 톤, 굵은 외곽선, 천진한 동물 캐릭터" 같은 스타일 디스크립터를 뽑음. 이 디스크립터를 페이지별 이미지 프롬프트에 강제로 주입.
- **이유**: gpt-image-1의 reference image 입력은 일관성이 들쭉날쭉. Vision으로 스타일을 텍스트화한 뒤 텍스트 프롬프트로 일러스트 생성하면 페이지 간 스타일 일관성이 더 잘 유지됨 + 디버깅 쉬움 + 비용 통제 가능.
- **스토리 생성**: GPT-4o-mini로 5페이지 동화 생성. 페이지당 [본문 2-3문장 + 일러스트 프롬프트] JSON 출력 강제.
- **이미지 생성**: 페이지별로 gpt-image-1 호출. 동일 스타일 디스크립터 prefix + 페이지별 프롬프트.

**데이터 모델 (JPA 필드명 — DB 컬럼은 snake_case로 매핑)**

패키지 그룹핑: 도메인 layer만 feature(`story`/`order`)로 한 번 더 나눔. 헥사고날·DDD 같은 무거운 구조는 도입 안 함 (30h 마감 + 면접 답변 부담). 자세한 컨벤션은 `docs/WORKTREES.md`.

```
com.sweetbook.domain.story
  Story
    id (UUID), title, childName
    status (DRAFT / ANALYZING_DRAWING / GENERATING_STORY / GENERATING_IMAGES / COMPLETED / FAILED)
    errorMessage (nullable, FAILED 시 한국어 사용자용 메시지)
    drawingUrl, styleDescriptorJson (JSON 컬럼)
    imaginationPrompt (text)
    createdAt, updatedAt

  Page
    id, storyId (FK), pageNumber (1~5)
    layout (COVER / SPLIT / ENDING)  -- pageNumber로 결정 (1=COVER, 2~4=SPLIT, 5=ENDING)
    bodyText (nullable, COVER는 제목으로 대체)
    illustrationPrompt, illustrationUrl (nullable, 생성 전엔 null)

  StoryStatus, PageLayout (enum)

com.sweetbook.domain.order
  Order  (Lv2)
    id (UUID), storyId (FK → domain.story.Story), recipientName, addressMemo
    status (PENDING / PROCESSING / COMPLETED)  -- 전진만 허용
    statusHistory (JSON list of {status, ts})
    createdAt, updatedAt

  OrderItem
    id, orderId (FK), bookSize (A5/B5), coverType (HARD/SOFT), copies (1~10)

  OrderStatus, BookSize, CoverType (enum)
```

**서비스 흐름 (Lv1)**

1. 메인: 동화 목록 (시드된 더미 동화 4편 즉시 노출)
2. 새로 만들기: 그림 업로드 → 상상 입력 → "동화 생성" 버튼 → 백엔드 비동기 처리(스토리 → 페이지별 이미지 → 완성)
3. 진행 상태: 클라이언트 폴링(`GET /api/stories/:id` 2초 간격)으로 페이지 1, 2, 3 ... 순차 노출
4. 완성: 5페이지 동화책 미리보기. 페이지 텍스트 인라인 편집 + 일러스트 재생성 버튼.

**주문 흐름 (Lv2)**

- 동화 상세에서 "이 동화로 책 만들기" → 주문 생성 모달 (사이즈/표지/부수)
- 주문 목록 페이지(`/orders`): status별 3컬럼 칸반 + 상태 변경 액션
- 상태 전이: PENDING → PROCESSING → COMPLETED. 전이 시 statusHistory append.

**데이터 익스포트 (가벼운 Lv3)**

- 엔드포인트: `GET /api/orders/{id}/export` → ZIP 응답
- ZIP 구조 (총 14개 엔트리: 4 + 5×2):
  ```
  order-{uuid}/
    metadata.json     (주문 정보 + 책 옵션 + statusHistory)
    story.json        (제목, 작가(아이 이름), pageCount)
    style.json        (분석된 스타일 디스크립터)
    drawing.png       (원본 아이 그림)
    pages/
      page-01.json    ({ pageNumber, layout, bodyText, illustrationPrompt, illustrationMissing })
      page-01.png     (일러스트, 실패 페이지면 placeholder.png 사본)
      page-02.json
      page-02.png
      ...
      page-05.json
      page-05.png
  ```
- 구현: ZipOutputStream + Jackson. 파일은 `Files.copy`로 stream에 직접 복사 (BLOB 비효율 회피).
- ZIP 안의 JSON 필드도 camelCase (API와 통일). 외부 분석 도구가 코드 매핑 없이 그대로 소비 가능.
- 인쇄용 PDF는 만들지 않음 (PDF가 명시).

**AI 키 부재 시 fallback**

- `OPENAI_API_KEY`가 비어 있으면 `application.yml`의 `app.ai.mock-mode=true` → mock 응답 (사전 준비된 동화 JSON + placeholder 이미지) 반환.
- 시드된 더미 동화 4편은 미리 생성한 실제 결과물 (txt+png)을 `resources/seed/`에 박아두고 마이그레이션 시 DB에 주입. 심사자가 키 없이도 서비스 흐름·주문·익스포트 모두 시연 가능.
- mock 모드는 각 단계(ANALYZING_DRAWING / GENERATING_STORY / GENERATING_IMAGES) 사이에 600ms sleep을 넣어 단계 전이를 흉내냄 → 데모 영상에서 진행 상태 UI 살아있게 보임.

## UX/UI Specification (2026-04-28 보강)

### 사이트맵

라우트 4개 + 모달 1개 + 다운로드 1개. 단일 사용자 가정(인증 없음).

```
/  (메인)                  ─→  /stories/new  ─→  /stories/:id  ──┬──→  주문 모달 (오버레이)
                                                                  │            │
/orders  (주문 칸반)  ←──────────────────────────────────────────┘            │
   └─→ GET /api/orders/:id/export  (ZIP 다운로드, 라우트 아님)                │
                                                                              ↓
                                                                         POST /api/orders
                                                                              ↓
                                                                         /orders로 이동
```

상단 네비: `📚 동화 목록` · `📦 주문` 두 항목.

### 상태 머신

**Story.status 전이 (전진만, FAILED는 재시도 시 ANALYZING_DRAWING으로 복귀)**

```
DRAFT
  ↓ 비동기 워커 시작
ANALYZING_DRAWING (GPT-4o Vision)
  ↓
GENERATING_STORY (GPT-4o-mini, JSON 출력)
  ↓
GENERATING_IMAGES (gpt-image-1 × 5, 병렬, 완료 페이지부터 점진 표시)
  ↓
COMPLETED  ←──┐
              │
FAILED ───────┘  POST /api/stories/:id/retry로 ANALYZING_DRAWING으로 복귀
```

- 페이지 5장 중 일부 일러스트만 실패해도 전체 status=COMPLETED 유지(부분 성공). 해당 페이지에 placeholder + 재생성 버튼.
- 스토리 생성 자체 실패(JSON 파싱 실패 등) 또는 그림 분석 실패 시만 status=FAILED.
- mock 모드는 각 단계에서 600ms sleep으로 단계 전이 흉내 (데모 임팩트).

**Order.status 전이 (전진만, 후진/스킵 시 409 INVALID_TRANSITION)**

```
PENDING ──→ PROCESSING ──→ COMPLETED
```

- `PATCH /api/orders/:id/status`에서 전이 검증. 위반 시 HTTP 409 + `{ error: "INVALID_TRANSITION" }`.
- 409(Conflict)는 "리소스는 존재하지만 현재 상태가 그 작업을 허용하지 않음"이라는 의미라서 400보다 정확. 구현은 `IllegalStateException` → `GlobalExceptionHandler` → `HttpStatus.CONFLICT`.
- 전이마다 `statusHistory` JSON에 `{ status, ts: ISO8601 }` append.

### 화면 1 — 메인 (`/`)

- 상단 hero("아이의 그림과 상상이 동화책이 됩니다") + "+ 새 동화 만들기" CTA.
- 4컬럼 카드 그리드. 카드 = 동화의 표지 페이지(pageNumber=1, layout=COVER) 그대로 노출.
- 카드 3가지 상태:
  1. **완성**: 표지 일러스트 + 제목 + "5페이지" 라벨.
  2. **생성중**: 진행률 바 + "3/5 페이지 생성중" + 카드 단위 폴링 2s.
  3. **FAILED**: 빨간 테두리 + ⚠️ 아이콘 + "다시 시도" 버튼.
- 빈 상태: 별도 화면 ("아직 동화가 없어요" + 첫 동화 만들기 CTA).
- 시드 동화 4편이 docker-compose up 직후 즉시 그리드를 채움.

### 화면 2 — 신규 동화 만들기 (`/stories/new`)

좌우 split 레이아웃: 좌측=그림 업로드 dropzone, 우측=아이 이름·상상 입력 + 제출.

**유효성 룰 (백+프론트 공통)**

| 필드 | 제약 | 위반 시 메시지 |
|------|------|----------------|
| `drawing` | image/jpeg, image/png · ≤ 5MB · ≥ 512×512px | "5MB 이하 JPG/PNG만 업로드 가능합니다" |
| `childName` | 필수 · 1~20자 · trim | "아이 이름은 1~20자로 적어주세요" |
| `imaginationPrompt` | 필수 · 10~500자 | "상상은 10자 이상 500자 이하로 적어주세요" |

제출 즉시 백이 `{ id, status: "DRAFT" }` 응답 → 프론트는 즉시 `/stories/:id`로 라우팅하고 폴링 시작.

### 화면 3 — 동화 미리보기 + 편집 (`/stories/:id`)

데모 메인 무대. 두 가지 뷰:

**생성중 뷰**: 4단계 step indicator(그림 분석 → 스토리 → 일러스트 → 완성) + 페이지 5장 진행률 그리드. 완료된 페이지부터 ✓로 표시.

**완성 뷰**:
- 상단 before/after 스트립: 원본 그림 썸네일 → 스타일 디스크립터 텍스트("수채화풍, 따뜻한 파스텔 톤..."). 면접 영상 한 컷의 핵심.
- 책장 형태 뷰어: 좌우 페이지 펼침. 좌/우 양 옆에 ‹ › 버튼 + 하단 5장 썸네일 스트립.
- 페이지별 편집 버튼: `✏️ 텍스트 수정`, `🔄 일러스트 재생성`. 레이아웃별로 노출 다름 (COVER는 제목·일러스트만, ENDING은 본문·일러스트).
- "📦 이 동화로 책 만들기" CTA → 주문 모달 오버레이.

**페이지 레이아웃 3종 (pageNumber로 하드코딩)**

| 페이지 | layout | 구성 |
|--------|--------|------|
| 1 | COVER | 전면 일러스트 + 제목 오버레이. 카드 썸네일과 동일 컴포넌트로 재사용. |
| 2~4 | SPLIT | 좌 일러스트 / 우 본문 (각 본문 2~3문장). |
| 5 | ENDING | 상단 본문 (마무리 한 줄) / 하단 일러스트. |

AI는 layout을 결정하지 않음 — 본문+이미지 프롬프트만 출력. layout은 프론트가 pageNumber로 컴포넌트 선택.

### 화면 4 — 주문 모달 (오버레이)

미리보기 화면 위에서 열림. 입력:
- 동화 요약 (제목 + 표지 썸네일, 읽기 전용)
- 책 사이즈: A5 / B5 (toggle)
- 표지: 소프트 / 하드 (toggle)
- 부수: 1~10 (stepper)
- 받는 분 이름 (필수, 1~30자)
- 주소 메모 (선택, 텍스트, 발송 안 함 명시)

제출 → `POST /api/orders` → `/orders`로 이동, 새 주문이 PENDING 컬럼에 노출.

### 화면 5 — 주문 관리 칸반 (`/orders`)

3컬럼 (PENDING / PROCESSING / COMPLETED). 컬럼 헤더에 카운트.

카드 구성:
- 동화 표지 썸네일 + 제목
- 옵션 한 줄 (예: "A5 · 하드 · 1부 · 받는분: 김서아")
- createdAt relative time
- 액션 버튼:
  - PENDING: `→ 제작 시작` (primary) + `📦 ZIP`
  - PROCESSING: `→ 완료` (primary, 초록) + `📦 ZIP`
  - COMPLETED: `📦 ZIP 다운로드` (primary)

상태 전이는 `PATCH /api/orders/:id/status` 한 번. 후진 불가.

## API Specification (2026-04-28 보강)

**컨벤션**

- JSON 응답·요청 본문, 멀티파트 폼 필드, ZIP 안의 JSON 모두 **camelCase** 단일 컨벤션.
- DB 컬럼은 SQL 표준에 맞춰 snake_case (예: `child_name` → JPA 엔티티의 `childName`에 `@Column(name="child_name")`로 매핑).
- Java 측은 `spring.jackson.property-naming-strategy` 기본값(LOWER_CAMEL_CASE) 유지. 별도 `@JsonProperty` 불필요.
- TS 타입(`frontend/src/types.ts`)도 동일 camelCase로 직접 매핑.

| Method | Path | 설명 |
|--------|------|------|
| GET | `/api/stories` | 목록. resp: `[{ id, title, childName, status, coverUrl, createdAt, errorMessage }]` (createdAt DESC) |
| GET | `/api/stories/{id}` | 상세 + 5페이지. 폴링 대상. resp: `{ id, title, childName, status, errorMessage, drawingUrl, styleDescriptor, imaginationPrompt, pages: [{ pageNumber, layout, bodyText, illustrationPrompt, illustrationUrl }], createdAt }` |
| POST | `/api/stories` | 신규. body: `multipart/form-data` (drawing + childName + imaginationPrompt). resp 즉시: `{ id, status: "DRAFT" }`. 비동기 시작. |
| PATCH | `/api/stories/{id}/pages/{n}` | 페이지 본문 수정. body: `{ bodyText }`. |
| POST | `/api/stories/{id}/pages/{n}/regenerate` | 해당 페이지 일러스트만 재생성. |
| POST | `/api/stories/{id}/retry` | FAILED 동화 재시도. status가 ANALYZING_DRAWING으로 복귀. |
| GET | `/api/orders` | 목록. resp: `[{ id, story: {id,title,coverUrl}, status, recipientName, addressMemo, item: {bookSize,coverType,copies}, createdAt }]` |
| POST | `/api/orders` | 신규. body: `{ storyId, bookSize, coverType, copies, recipientName, addressMemo }`. resp: `{ id, status: "PENDING" }`. |
| PATCH | `/api/orders/{id}/status` | 전이. body: `{ status }`. 위반 시 409 + `{ error: "INVALID_TRANSITION" }`. |
| GET | `/api/orders/{id}/export` | ZIP 응답. `Content-Disposition: attachment; filename=order-{uuid}.zip`. ZipOutputStream 스트리밍. |
| GET | `/api/files/{kind}/{filename}` | 이미지 서빙. kind ∈ {drawings, illustrations}. Spring `ResourceHandler`로 파일시스템 매핑. |

**공통 에러 포맷**: `{ "error": "ERROR_CODE", "message": "사용자용 한국어 메시지" }` + HTTP status (400 검증, 404 not found, 409 INVALID_TRANSITION, 500 서버, 503 AI 타임아웃).

**SPA 라우팅 fallback**: Vue Router는 history mode 사용 (createWebHistory). 백엔드는 `/api/**`·`/assets/**`·확장자 있는 경로 외 모든 GET을 `index.html`로 forward → 직접 진입·새로고침 시에도 `/stories/:id`, `/orders` 등이 살아있음 (`WebConfig#addViewControllers`).

## Image Storage Strategy (2026-04-28 보강)

**파일시스템 + Docker 볼륨** 채택.

```
uploads/
  drawings/{uuid}.{jpg|png}                # POST /api/stories 업로드본
  illustrations/{storyId}/page-{n}.png     # gpt-image-1 결과
  seed/                                    # 시드 동화 정적 자산
```

- DB는 path만 저장 (`varchar(255)`). 파일 바이트는 DB에 안 들어감.
- `application.yml`: `app.upload-dir=/data/uploads`.
- `docker-compose.yml`: `volumes: ["./uploads:/data/uploads"]`. 컨테이너 재시작에도 영속.
- Spring `WebMvcConfigurer`의 `addResourceHandlers`로 `/api/files/**` → `file:/data/uploads/`. 별도 컨트롤러 불필요.
- 시드: `resources/seed/`의 PNG들을 `ApplicationRunner`에서 `/data/uploads/seed/`로 복사 (1회만). DB 시드 INSERT 시 path는 `seed/...`로.
- ZIP 익스포트: 파일 시스템 path를 알고 있으니 `ZipOutputStream`에 `Files.copy`만 하면 됨. 메모리 효율적.

**거부된 대안**:
- BLOB: ZIP 시 메모리에 25MB 한 번에 올라감. DB 무거워짐.
- MinIO: 컨테이너 추가 + SDK 의존성. 30h 마감엔 사치.
